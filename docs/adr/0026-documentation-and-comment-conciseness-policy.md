# ADR-0026: ドキュメント・コメント簡潔性ポリシー — ADR 索引の最小化とコード内 ADR 参照の排除

- Status: Accepted
- Date: 2026-05-29
- Decider: -

---

## Context

設計判断の SSoT(Single Source of Truth)であるべき ADR 本体の内容が、索引とコードコメントに重複ミラーされ、二重・三重管理になっている実態が、整備の過程で定量的に確認された。

### 観測された肥大化

**1. ADR 索引の1行詰め込み**

- `docs/adr/README.md` の概要列は、1 セルに最長 1,328 字を詰め込んでいる。
- 上位 8 件が 859〜1,328 字。
- CLAUDE.md の ADR 一覧表も最大 365 字/行。
- 結果として「一覧して見渡す」という索引本来の機能を失っている。

**2. コード内 ADR 参照の氾濫**

- `Assets/_Project/Scripts` の `.cs` のうち 780 箇所 / 174 ファイル(全 250 ファイルの 70%)が `ADR-NNNN` を参照。
- 最長 345 字の設計経緯がコメント 1 行に詰め込まれている。
- 例: `// ADR-0014: StartGameUseCase は ICardCatalog<IEffect> 依存を削除済 → ADR-0024 で再追加 …`
- コードを読むたびに設計の歴史が視界に入り、コメント本来の役割(技術的な「なぜ / 何を」)が埋もれる。

### 根本原因

- Phase 1〜2 を通じて ADR が 25 本に成長し、各 ADR が長大な経緯を持つに至った。
- 「経緯を全部書く」習慣が索引・コメント双方に蓄積した。
- ADR 本体に書くべき内容が、索引列とコードコメントに重複コピーされ続けた。
- CLAUDE.md の ADR 運用節は既に「各 ADR を SSoT として参照する(CLAUDE.md ではミラーしない)」と表明済。本 ADR はその原則を索引列とコードコメントにも一貫適用するもので、新方針ではなく既存原則の徹底である。

## Decision

### 1. ADR 索引・ミラー類は「番号 + タイトル + Status」のみ

- `docs/adr/README.md` のインデックス表は番号 / タイトル / Status の 3 列のみとする(概要列を持たない)。
- CLAUDE.md §11・README.md など ADR をミラーする箇所も同様(3 要素、または委譲リンクのみ)。
- ADR の概要・経緯・詳細はすべて各 ADR 本体の Context が SSoT。
- 1 テーブルセル・1 行に長文(目安: 全角 120 字超)を詰め込まない。

### 2. コード内コメントに ADR 番号・経緯を書かない

- 通常コメント(`//`)・XML doc コメント(`///`)の双方で、ADR 番号(`ADR-NNNN`)と設計経緯を書かない。
- コメントは技術的な「なぜ / 何を / どうやって」に絞る。
- 設計判断の経緯・履歴の SSoT は ADR 本体。
- コード ↔ ADR の紐付けが必要なときは git blame / commit message / PR から辿る。

### 3. 適用範囲と除外

| 対象 | 本ポリシーの適用 |
| ---- | ---- |
| `Assets/_Project/Scripts` 配下の `.cs`(プロダクション + テスト) | ポリシー 2 を適用(ADR 参照を排除) |
| `docs/adr/README.md` / CLAUDE.md / README.md の ADR ミラー | ポリシー 1 を適用(番号 + タイトル + Status) |
| コミットメッセージ・PR 本文 | 対象外。従来どおり ADR 番号で相互参照してよい(推奨) |
| `docs/` 配下の Markdown(ADR 本体・仕様・設計文書) | 対象外。ADR 番号での相互参照を維持 |

### 4. 既存資産への適用方法

- 既存コードの ADR 参照除去は、本 ADR マージ後に層ごと(Domain → Application → Infrastructure → Presentation → Bootstrap → Tests)段階実施する。
- コメント本体は残し、ADR 番号・経緯の部分のみを削る(コメントごと消すのではない)。
- 各段階で `dotnet build` と EditMode テストで回帰を確認する。

## Consequences

### Positive

- 索引が一覧として機能する(全 26 行を一目で見渡せる)。
- コードコメントが技術情報に集中し、可読性が向上する。
- SSoT の一元化により、ADR 本体・索引・コメントの三重管理が解消される。
- 新規 ADR / カード追加時に「索引概要を書く」「コメントに経緯を書く」作業が不要になる。

### Negative

- コード単体から「なぜこの設計か」を辿るには git blame / PR を経由する一手間が増える。
- 既存 174 ファイル・780 箇所の除去作業コストが発生する(段階実施で緩和)。
- ADR の概要を素早く知りたいとき、索引では分からず本体を開く必要がある(本体冒頭の Context が実質サマリの役割)。

### Neutral

- ADR 本体の書式(必須 4 セクション等、ADR-0001)は不変。
- コミットメッセージへの ADR 番号付与規約は不変。

## Related

- ADR-0001 の運用(`docs/adr/README.md` の索引・起票手順)を 3 列構成に具体化した(Supersede せず拡張)。
- CLAUDE.md の言語規約に「コメントに ADR 番号・経緯を書かない」を追加。
- CLAUDE.md(ADR 運用節)/ README.md の ADR ミラーを本ポリシーに整合。
- 関連実装:
  - `docs/adr/README.md` 索引の 3 列圧縮。
  - コード内 ADR コメントの一括除去(層ごと段階実施)。
